USB-C Cable Orientation — READ FIRST

The board uses a CH445P analog switch to multiplex a single USB-C port between two MCUs. The physical orientation of the USB-C plug determines which MCU gets programmed.

MUST use USB-A to USB-C cable!

USB-C to USB-C cables do NOT reliably select the MCU — the CC pin negotiation is unpredictable. Use a USB-A to USB-C cable where flipping the plug at the USB-C end switches the MCU.

USB-C OrientationRoutes ToPort Type
Orientation AESP32-S3 native USB (GPIO19/20) — THIS IS WHAT WE WANT/dev/cu.usbmodem* or /dev/ttyACM*
Orientation B (flip 180deg)ESP32 via CH343P bridge (UART0)/dev/cu.wchusbserial* or /dev/ttyUSB*

How to verify which MCU you're talking to:

  1. Plug in USB-A to USB-C cable
  2. Run ls /dev/cu.usb* (macOS) or ls /dev/ttyACM* /dev/ttyUSB* (Linux)
  3. Run esptool.py --port /dev/cu.XXXXX chip_id
  4. If it reports ESP32-S3 -> correct orientation
  5. If it reports ESP32 -> flip the USB-C plug 180deg and retry

If no port appears:

  1. macOS: Install CH343 macOS driver, authorize in System Settings > Privacy & Security
  2. Hold BOOT button -> Press RST -> Release BOOT (enters download mode)
  3. Check ports again

Boot mode sequence: Hold BOOT -> Press RST -> Release BOOT. The auto-download circuit (DTR/RTS) usually handles this automatically during idf.py flash.

Firmware Programming Guide

Step 1: Environment Setup

# Install ESP-IDF (if not already installed)
# Follow: https://docs.espressif.com/projects/esp-idf/en/stable/esp32s3/get-started/

# Verify installation
idf.py --version

# Clone this repo (if not already)
git clone https://github.com/amok-products/strong-vibes-knob-control.git
cd strong-vibes-knob-control

Step 2: Configure and Build

# Set target to ESP32-S3
idf.py set-target esp32s3

# Build the firmware
idf.py build

# If build fails with missing components, install them:
idf.py add-dependency "espressif/esp_lcd_st77916"
idf.py add-dependency "espressif/esp_lvgl_port"
idf.py add-dependency "lvgl/lvgl"

Step 3: Flash

Before Flashing

1. Plug USB-C cable with correct orientation (see USB-C section above)
2. Verify serial port: ls /dev/cu.usb*
3. If port not found: Hold BOOT -> Press RST -> Release BOOT

# Flash and open serial monitor
idf.py -p /dev/cu.usbmodem* flash monitor

# macOS: port is usually /dev/cu.usbmodemXXXX
# Linux: port is usually /dev/ttyACM0
# Windows: COM port (check Device Manager)

# If flash fails, enter download mode:
# Hold BOOT button -> Press RST button -> Release BOOT button
# Then retry idf.py flash

# Exit monitor: Ctrl+]

Step 4: Verify

# Monitor should show:
# I (XXX) knob-ctl: Strong Vibes Knob Control starting...
# I (XXX) display: Initializing ST77916 LCD...
# I (XXX) display: Display ready (360x360)
# I (XXX) encoder: Encoder on GPIO8/7 (knob component), range 0-100, step 2
# I (XXX) knob-ctl: Strong Vibes Knob Control initialized

# If you see garbage or nothing:
# 1. Check baud rate: idf.py -p PORT monitor -b 115200
# 2. Check cable orientation (flip USB-C if needed)
# 3. Press RST button to reboot

Display Output

Status Screen (360x360 ST77916 LCD)

The LVGL-based status screen shows real-time device state:

ElementDescriptionUpdates When
MIC Arc Gauge270-degree green arc showing MIC intensity 0-100Rotary encoder turned
MIC ValueLarge number (Montserrat 28pt) centered in arcRotary encoder turned
Webapp Status"Webapp: --" (gray) or "Webapp: Connected" (green)BLE server connect/disconnect
EMW Status"EMW: --" (gray) or "EMW: Connected" (green)BLE client connect/disconnect
BLE Activity"BLE" label flashes blue for 300msDatagram forwarded through bridge

The panel is driven over QSPI by the native espressif/esp_lcd_st77916 driver. On hardware the display renders correctly — colors, orientation, and backlight are all good, with no faint center line observed across cold boots. (An earlier revision drove the panel through the esp_lcd_sh8601 component as a generic QSPI shim; the panel receives the same register stream under either driver, so the driver identity is a correctness matter, not a display-behavior one.)

BLE Protocol (2-Char Design)

RoleServiceChar 1225 (Vibe Write)Char 1227 (Vibe Read)
Server (for Webapp)0x1221Receives datagrams + opcodesSends notifications
Client (to EMW)Connects to 0x1221Writes datagrams + opcodesSubscribes to notifications

Discrimination: byte[0] == 0x10 -> datagram, else -> raw opcode
