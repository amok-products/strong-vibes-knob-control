# Vibe Protocol — Implementation Reference (public)

The **Vibe Protocol** lets a *client* (a browser app, a mobile app, or an LLM/agent) drive motor/vibration patterns on a *server* device (e.g. the Europe Magic Wand "EMW", or a Knob bridge) over Bluetooth Low Energy. It is a compact, datagram-based protocol carried on **two GATT characteristics**. Everything fits in a single characteristic write or notification — no multi-segment reassembly is required at the transport level (though long patterns can be split into multiple datagrams, §11).

This document is **self-contained**: an AI or developer can read it once and build a working Chrome/Edge Web Bluetooth controller that connects to a Vibe device and runs patterns. It describes only the Vibe Protocol — nothing about how the device works internally.

> **Vibe is not alone on the wire.** A Vibe server may also speak a separate, private protocol for its own companion app over the *same* two characteristics, discriminated by the first byte of each message. The Vibe first byte is always `0x10`. **Treat the following as reserved — never send them as a Vibe message first byte:** `0x80`, `0x81`, and the byte range `0x01..0x0F`. As long as your messages start with `0x10`, the two protocols coexist cleanly and the other protocol's traffic is invisible to you. (If you sniff raw traffic and see frames that do not start with `0x10` or `0xFF`, that is the other protocol — ignore it.)

---

## 1. Architecture: roles

- **Server** — the device that advertises the Vibe Service and runs motor patterns (EMW, Knob, any future Vibe device).
- **Client** — your app. It scans, connects, discovers the service, and sends control datagrams; it subscribes for state/error notifications.

The client is an accessory, not a master: a well-behaved Vibe device **keeps running its last pattern if the BLE link drops** and reports its current state when a client (re)connects. Reflect the device's reported state in your UI on reconnect — don't blindly push your cached state back to the device.

---

## 2. BLE service & characteristics

Base UUID pattern — substitute the 4-hex short UUID for `xxxx`:

```
0000xxxx-1313-71be-1221-785dfeabc321
```

| Short UUID | Full UUID | Name | Properties | Direction |
|---|---|---|---|---|
| `1221` | `00001221-1313-71be-1221-785dfeabc321` | Vibe Service | — | service (discovery) |
| `1225` | `00001225-1313-71be-1221-785dfeabc321` | **Vibe Write** | Write, Write-No-Response | client → server |
| `1227` | `00001227-1313-71be-1221-785dfeabc321` | **Vibe Read** | Read, Notify | server → client |

All client→server data goes to **1225**; all server→client data arrives as notifications on **1227**. Both are control points (values consumed immediately, not stored). There are no other active characteristics — older drafts had 1222–1230 but those are removed (§12).

---

## 3. Transport & data size

- **BLE 4.2+** only. All data fits in single characteristic operations.
- **Max datagram = 121 bytes.** A common Vibe module (the PB-03 on the EMW) caps the usable ATT payload at 121 bytes (`L2CAP_MTU=128 → ATT_MTU=124 → ATT payload=121`). Stay at or below 121 bytes per write/notify.
- **Max datagram payload = 119 bytes** (121 − 2 header bytes).

**Discrimination — Vibe Write (1225):** the server reads `byte[0]`:
- `0x10` → **Vibe datagram** (the normal case; see §4).
- anything else → not a Vibe datagram (reserved for the device's other protocol — don't send it).

**Discrimination — Vibe Read (1227):** the client reads `byte[0]`:
- `0x10` → datagram-wrapped response (Motor.State, Motor.ErrorEvent, Device.Info response).
- `0xFF` → error response: `[0xFF, error_code, original_first_byte]`.

> **Fragmentation on notify (real-world):** some Vibe modules emit a long datagram as *several* small GATT notifications (~8 bytes each). The browser delivers each as its own `characteristicvaluechanged` event. If you query large responses (e.g. the Device.Info bundle, §9), reassemble by buffering on the `[0x10, SIZE]` header until `SIZE+2` bytes have arrived (with a ~200 ms stale-flush). Small datagrams (Motor.State = 7 bytes) arrive in one notify and don't need this.

---

## 4. Vibe datagram format

Every datagram is:

```
| VER (1 byte) | SIZE (1 byte) | PAYLOAD (0..119 bytes) |
```

- **VER** = `0x10` (protocol v1.0, also the discrimination byte).
- **SIZE** = number of payload bytes that follow (0..119). It does **not** count VER or SIZE.
- **PAYLOAD** = one Vibe module message (§5).

Total on-wire bytes = `2 + SIZE` (≤ 121).

---

## 5. Modules & messages

The payload's first two bytes are `[Module ID][Message ID]`.

| Module ID | Module | Messages |
|---|---|---|
| `0x00` | Control | `0x00` Reset |
| `0x01` | Motor | `0x00` RepeatingPattern, `0x01` NonRepeatingPattern, `0x02` State (notify), `0x03` ErrorEvent (notify), `0x04` MainIntensityControl |
| `0x03` | Device.Info | slot query/response/write (§9) |

(Module `0x02` is reserved — see §10. Modules `0x04..0x7F` are reserved for future expansion, §13.)

### 5.1 Control.Reset — Module `0x00`, Message `0x00`

Stops any running pattern and returns the server to idle.

```
Payload:  [0x00, 0x00]
Datagram: [0x10, 0x02, 0x00, 0x00]
```

### 5.2 Motor module — Module `0x01`

#### Timbre (waveform) IDs

| Timbre | Waveform | Segment size |
|---|---|---|
| `0x00` | Linear transition | 4 bytes / segment |
| `0x01` | Sine wave | 4 bytes (single parameter set) |
| `0x02` | Gaussian transition | 5 bytes / segment |

#### Motor.RepeatingPattern — Message `0x00`

Plays a pattern that loops until replaced or reset.

**Header (5 bytes):**

```
[0x01]        Module: Motor
[0x00]        Message: RepeatingPattern
[Timbre]      0x00 Linear | 0x01 Sine | 0x02 Gaussian
[Pattern ID]  client-assigned 1..254 (used to correlate Motor.State)
[MIC]         Main Intensity Control 0..100 (or special, see Motor.MIC)
```

…followed by the waveform data:

- **Linear segment (4 bytes each)** — `[IntensityBegin(0-100)][IntensityEnd(0-100)][Period_hi][Period_lo]`. Period is uint16 milliseconds, **big-endian**. The motor ramps linearly Begin→End over Period, then advances; after the last segment it loops.
- **Sine parameters (4 bytes, one set)** — `[Amplitude(0-100)][Center(0-100)][Period_hi][Period_lo]`. Output ≈ `Amplitude·sin(2π·t/Period) + Center`.
- **Gaussian segment (5 bytes each)** — `[StartPoint(0-100)][EndPoint(0-100)][Duration_hi][Duration_lo][Steep]`. Output ≈ `(StartPoint − EndPoint)·e^(−t²/(2·Steep²)) + EndPoint` (Steep unit ≈ 20 ms). Multiple segments chain, then loop.

#### Motor.NonRepeatingPattern — Message `0x01`

Identical format to RepeatingPattern but plays **once** then stops:

```
[0x01, 0x01, Timbre, PatternID, MIC, ...segments]
```

#### Motor.State — Message `0x02` (server notification on 1227)

```
[0x01, 0x02, State, PatternID, MIC]
```

| State | Meaning |
|---|---|
| `0` | Idle — nothing running |
| `1` | Running a repeating pattern |
| `2` | Running a non-repeating pattern |
| `3` | Error |

`PatternID`: `0x00` = idle/error, `0xFF` = server-dictated pattern, `1..254` = the client-assigned ID you sent. Use the running→idle **transition** of `State` to sequence multi-datagram patterns (§11).

#### Motor.ErrorEvent — Message `0x03` (server notification on 1227)

```
[0x01, 0x03, ErrorCode]
```

| ErrorCode | Meaning |
|---|---|
| `1` | Motor error |
| `2` | Communication error |
| `3+` | Reserved |

#### Motor.MainIntensityControl — Message `0x04`

Adjusts the global intensity multiplier on the fly without resending the pattern.

```
[0x01, 0x04, MIC]
Datagram: [0x10, 0x03, 0x01, 0x04, MIC]
```

| MIC value | Meaning |
|---|---|
| `0x00..0x64` (0–100) | set intensity percentage |
| `0xEE` (238) | lock at 100 % until the pattern stops |
| `0xFF` (255) | leave unchanged (no-op) |

Effective output = `MIC · segment_intensity(t)` — MIC is a global pre-gain.

---

## 6. Pattern interaction rules

**While a repeating pattern runs:**

| Incoming | Behaviour |
|---|---|
| new repeating | replaces the current pattern immediately |
| non-repeating | interrupts, plays once, then resumes the repeating pattern |

**While a non-repeating pattern plays:**

| Incoming | Behaviour |
|---|---|
| repeating | queued — runs after the non-repeating finishes |
| non-repeating | rejected |

The client is responsible for queuing/sequencing if it needs more (§11).

---

## 7. Max segments per datagram

119 payload bytes − 5-byte motor header = 114 bytes for segments:

| Timbre | Segment size | Max segments / datagram |
|---|---|---|
| Linear `0x00` | 4 bytes | 28 |
| Sine `0x01` | 4 bytes | 1 (single param set) |
| Gaussian `0x02` | 5 bytes | 22 |

Longer patterns use client-side multi-datagram chunking (§11).

## 8. Quick reference: building messages

**Linear repeating pattern**

1. Build segments, each `[IntBegin, IntEnd, PeriodHi, PeriodLo]`.
2. Header `[0x01, 0x00, 0x00, PatternID, MIC]`.
3. Wrap `[0x10, SIZE, ...header, ...segments]` and write to **1225** in one operation.

**Example — 4-segment linear repeating pattern (ID=1, MIC=100 %)**

```
Header: [0x01, 0x00, 0x00, 0x01, 0x64]      Motor, Repeating, Linear, ID=1, MIC=100%
Seg 1:  [0x00, 0x64, 0x07, 0xD0]            0% → 100% over 2000 ms
Seg 2:  [0x64, 0x64, 0x03, 0xE8]            hold 100% for 1000 ms
Seg 3:  [0x64, 0x00, 0x07, 0xD0]            100% → 0% over 2000 ms
Seg 4:  [0x00, 0x00, 0x03, 0xE8]            hold 0% for 1000 ms

Datagram (SIZE = 0x15 = 21 bytes = 5 header + 16 segments):
[0x10,0x15, 0x01,0x00,0x00,0x01,0x64, 0x00,0x64,0x07,0xD0, 0x64,0x64,0x03,0xE8,
 0x64,0x00,0x07,0xD0, 0x00,0x00,0x03,0xE8]
```

**Adjust MIC live:** `[0x10, 0x03, 0x01, 0x04, MIC]`
**Reset motor:** `[0x10, 0x02, 0x00, 0x00]`
**Read firmware version:** query Device.Info slot `0x05` (§9) — *not* a raw opcode (§10).

## 9. Device.Info — Module `0x03`

Read-only telemetry plus one writable settings slot: firmware version, hardware revision, extended serial number, fault counters, and (where the hardware supports them) lifetime usage and battery. The client queries; the server replies on 1227. There are no unsolicited Module-`0x03` emissions.

**Per-slot query:** `[0x10, 0x02, 0x03, SLOT_ID]`
**Bundle query (all populated slots):** `[0x10, 0x02, 0x03, 0xFF]`

**Per-slot response:** `[0x10, SIZE, 0x03, SLOT_ID, SCHEMA_VER, ...payload]`
**Bundle response:** `[0x10, SIZE, 0x03, 0xFF, <sub-record>, <sub-record>, ...]` where each sub-record = `[SLOT_ID, SUB_LEN, SCHEMA_VER, ...body]` and `SUB_LEN` counts `SCHEMA_VER + body` (so a sub-record is `2 + SUB_LEN` bytes on the wire). Iterate the bundle by length, not by expected slot.

| SLOT_ID | Name | Payload (after SCHEMA_VER) | Notes |
|---|---|---|---|
| `0x01` | Battery | `[pct][mV_hi][mV_lo][flags]` | **optional** — absent on hardware without battery sense; **NA (not populated) on the reference EMW firmware** |
| `0x02` | Fault | `[active][last][count_hi][count_lo]` | |
| `0x03` | HW revision | `[pcb][motor][feat_hi][feat_lo][0,0,0,0]` | `pcb`/`motor` are nibble-decimal (`0x30`→`3.0`, `0x10`→`1.0`) |
| `0x04` | Usage | `[toggles:3 BE][runtime_minutes:4 BE]` | **optional** — **NA (not populated) on the reference EMW firmware** |
| `0x05` | FW version | `[major][minor][patch][build][git:4 BE]` | render `major.minor.patch.build` |
| `0x06` | Device settings | `[settings_byte]` (bit 0 = a device-specific toggle; bits 1–7 reserved) | **writable** |
| `0xFE` | Extended SN | `[SN:4 BE]` | bundle / per-slot only |
| `0xFF` | Bundle marker | query side only; response carries sub-records | |

All multi-byte fields are big-endian. `SCHEMA_VER` is currently `0x01` for every slot.

**Optional slots.** Battery (`0x01`) and Usage (`0x04`) are optional: a device populates them only if it has the corresponding hardware/telemetry. Neither is populated by the reference EMW firmware, so on that hardware they simply never appear in a bundle or per-slot response — this is expected, not an error. A client must treat *any* slot as possibly-absent (drive the UI off the slots that actually arrive) and never require `0x01`/`0x04`.

**Schema evolution (append-only):** the leading `SCHEMA_VER` announces a slot's layout; older clients parse what they understand and **ignore trailing bytes** and **skip unknown sub-slots** (advance `SUB_LEN + 2`). Never reject a slot/bundle just because it is longer or has an unknown slot — future servers add fields and slots.

**Write (slot `0x06` only):**

```
Write: [0x10, 0x04, 0x03, 0x06, 0x01, new_settings_byte]
Echo:  [0x10, 0x04, 0x03, 0x06, 0x01, device_confirmed_byte]   (on 1227)
```

Compare the echoed byte against your requested value to confirm. Slots `0x01–0x05`, `0xFE` are read-only.

**Pull-on-connect (recommended client behaviour):** wait ≥ 500 ms after enabling notifications (some modules need their notify queue to drain), send the bundle query, retry once after 2 s, then fall back to per-slot pulls (`0x02 → 0x03 → 0x05 → 0xFE`) with ≥ 300 ms gaps (skip the optional `0x01`/`0x04` slots on devices that don't populate them — e.g. the reference EMW firmware). Per-slot is more resilient under a bad link; the bundle is one round-trip but fragments into several notifies.

## 10. Reserved / not part of v1.0

These appear in older drafts but are **not implemented** on the current reference firmware — do not depend on them:

- **Raw opcodes `0x0F` (query version), `0x0B` (query status), `0x0A` (sync).** The current EMW firmware does not handle non-`0x10` Vibe first bytes. **Get the firmware version from Device.Info slot `0x05` instead** (§9).
- **Authentication (Module `0x02`).** An MD5-style challenge/response was reserved for a future Module `0x02`; it is not implemented. Vibe today has no application-layer auth.

If you are writing against a *different* Vibe server (e.g. a Knob bridge) that advertises these, treat them as server-specific extensions — but a portable Vibe client should rely only on §4–§9.

## 11. Long patterns: client-side multi-datagram chunking

When a pattern needs more segments than fit in one datagram (>28 linear, >22 gaussian), split it client-side into self-contained datagrams and sequence them with Motor.State:

1. Send chunk 1 as **NonRepeatingPattern**.
2. The server sends `Motor.State` `State=2` (running non-repeating); record "running".
3. When the chunk finishes the server sends `Motor.State` `State=0` (idle).
4. On the **running→idle transition** (guard with a `wasRunning` flag — not just `State==0`, which also fires on stale idle), send the next chunk as NonRepeatingPattern.
5. Repeat; to loop, restart from chunk 1 after the last chunk's idle.

Notes:
- Drive sequencing from your **chunk queue**, not a UI "isPlaying" flag (auto-stop-on-idle logic races chunk advancement).
- There is a brief gap (~100–300 ms: idle notify + round-trip + startup) between chunks. Correct the visual playhead by snapping `playbackStartTime = now − Σ(periods of completed chunks)` when `State > 0`. Bridging the *motor* gap is a server responsibility (a hold-at-last-intensity window).
- Wait for the idle notification before sending the next chunk — the server processes one pattern at a time.

## 12. Legacy characteristics (removed)

Not used in the 2-characteristic design; listed so you ignore them if a device still exposes them. All functionality is multiplexed onto 1225 (write) / 1227 (notify):

`1222` (Version), `1223`/`1224` (auth challenge/response), `1226`/`1228` (BLE-4.1 segmentation buffers), `1229` (client command), `1230` (server state).

---

## 13. Reserved ranges for future expansion

To keep Vibe forward-compatible and collision-free with the device's other protocol, treat these as reserved:

| Field | Used today | Reserved for future |
|---|---|---|
| Datagram VER / first byte | `0x10` | nothing else — exactly `0x10` |
| Module ID | `0x00` Control, `0x01` Motor, `0x03` Device.Info | `0x02` (auth), `0x04..0x7F`; keep `0x80..0xFF` for system/extended |
| Motor (Module `0x01`) Message ID | `0x00,0x01,0x02,0x03,0x04` | `0x05..0x7F` |
| Device.Info (Module `0x03`) slot/Message ID | `0x01..0x06,0xFE,0xFF` | `0x07..0xFD` |
| First bytes the device reserves for its *other* protocol | — | `0x01..0x0F`, `0x80`, `0x81` — never use as a Vibe first byte |

When adding a feature, prefer a new Message under an existing Module (or a new Device.Info slot) over a new top-level Module.

## 15. Web Bluetooth quick-start (Chrome / Edge)

A minimal client. Web Bluetooth requires a secure context (HTTPS or `localhost`) and a user gesture to call `requestDevice`.

```js
const SVC   = '00001221-1313-71be-1221-785dfeabc321';
const WRITE = '00001225-1313-71be-1221-785dfeabc321';
const NOTIFY= '00001227-1313-71be-1221-785dfeabc321';

// 1. Connect (must be called from a click handler).
const device = await navigator.bluetooth.requestDevice({
  filters: [{ services: [SVC] }],            // or namePrefix: 'EMW_'
  optionalServices: [SVC],
});
const server = await device.gatt.connect();
const svc    = await server.getPrimaryService(SVC);
const writeCh  = await svc.getCharacteristic(WRITE);
const notifyCh = await svc.getCharacteristic(NOTIFY);

// 2. Subscribe to server → client notifications (Motor.State, ErrorEvent, Device.Info).
await notifyCh.startNotifications();
notifyCh.addEventListener('characteristicvaluechanged', (e) => {
  const b = new Uint8Array(e.target.value.buffer);
  if (b[0] === 0xFF) { console.warn('Vibe error', b[1], 'on', b[2]); return; }
  if (b[0] === 0x10 && b[2] === 0x01 && b[3] === 0x02) {
    console.log('Motor.State', { state: b[4], patternId: b[5], mic: b[6] });
  }
  // b[0]===0x10 && b[2]===0x03 → Device.Info response (reassemble if fragmented)
});

// 3. Helper: write a datagram.
const send = (payload) =>
  writeCh.writeValueWithoutResponse(new Uint8Array([0x10, payload.length, ...payload]));

// 4. Run a 4-segment linear repeating pattern, ID=1, MIC=100%.
await send([0x01,0x00,0x00,0x01,0x64,
            0x00,0x64,0x07,0xD0,  0x64,0x64,0x03,0xE8,
            0x64,0x00,0x07,0xD0,  0x00,0x00,0x03,0xE8]);

// 5. Live intensity, reset, version query.
await send([0x01,0x04,0x32]);          // MIC = 50%
await send([0x00,0x00]);               // Control.Reset
await send([0x03,0x05]);               // Device.Info: query FW version (slot 0x05)
```

Implementation tips:
- Use `writeValueWithoutResponse` for control-point writes; the device consumes them immediately.
- Keep every write ≤ 121 bytes; split longer patterns (§11).
- Handle the fragmented-notify case (§3) when you query the Device.Info bundle.
- On `gattserverdisconnected`, the device keeps running — re-read its state via Device.Info / Motor.State on reconnect rather than re-pushing your last command.

## 16. Server implementation requirements (if you build a Vibe device)

- Two characteristics only: 1225 (write) and 1227 (notify). Discriminate inbound by `byte[0]` (`0x10` = Vibe).
- Maintain two pattern buffers (active repeating + non-repeating) and the interaction rules of §6.
- Emit `Motor.State` on every state transition (especially pattern-complete → idle, so clients can chunk).
- Send all responses/notifications on 1227, one datagram per notification.
- On disconnect/link-loss: keep the motor running (autonomy), reset only the per-connection buffering.
